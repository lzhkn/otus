Что мы построили и как это работает
Общая идея
Мы построили сеть из трёх офисов, соединённых через центральный роутер, с единой точкой выхода в интернет. Это классическая корпоративная топология — hub-and-spoke.
Internet → inetRouter → centralRouter → office1Router → серверы office1
                                      → office2Router → серверы office2
                                      → centralServer

Уровни сети
Транзитные линки /30 — соединяют роутеры между собой. Маска /30 даёт всего 2 хоста — ровно столько нужно для линка между двумя роутерами. Никаких лишних адресов.
inetRouter .1 ←→ .2 centralRouter      (192.168.255.0/30)
centralRouter .9 ←→ .10 office1Router  (192.168.255.8/30)
centralRouter .5 ←→ .6 office2Router   (192.168.255.4/30)
Офисные сети /26 и /25 — конечные сети где живут серверы. Размер подбирался под задание.
mgmt-сеть 192.168.0.0/24 — отдельная сеть Proxmox для управления ВМ по SSH. Не участвует в лабораторном трафике.

Как работает маршрутизация
Каждый хост знает только куда отправить пакет — не весь путь целиком, а только следующий хоп.
Например, office1Server хочет пинговать 8.8.8.8:
office1Server (192.168.2.130)
  → default via 192.168.2.129 (office1Router)

office1Router (192.168.255.10)
  → default via 192.168.255.9 (centralRouter)

centralRouter (192.168.255.2)
  → default via 192.168.255.1 (inetRouter)

inetRouter
  → default via 192.168.0.32 (провайдер)
  → NAT MASQUERADE → пакет уходит в интернет
Ответ идёт обратно — и вот тут важны обратные маршруты.

Обратные маршруты — самое важное
Когда ответ от 8.8.8.8 возвращается на inetRouter, тот должен знать куда его отправить. Поэтому:
На inetRouter прописан маршрут:
192.168.0.0/16 via 192.168.255.2
Весь приватный диапазон — через centralRouter.
На centralRouter прописаны маршруты к офисам:
192.168.2.0/24 via 192.168.255.10  → в office1
192.168.1.0/24 via 192.168.255.6   → в office2
Без этих маршрутов centralRouter не знал бы куда отправить ответный пакет и дропал бы его.

Как работает NAT
На inetRouter одно правило iptables:
-A POSTROUTING ! -d 192.168.0.0/16 -o ens18 -j MASQUERADE
Когда пакет из 192.168.2.130 уходит в интернет через ens18 — inetRouter подменяет source IP на свой (192.168.0.130). Интернет отвечает на 192.168.0.130, inetRouter восстанавливает оригинальный адрес и отправляет ответ обратно. Серверы в офисах не имеют публичных IP — они все прячутся за одним адресом inetRouter.
Условие ! -d 192.168.0.0/16 исключает внутренний трафик — пакеты между серверами не маскируются.

IP forwarding — почему он нужен
По умолчанию Linux не пересылает пакеты адресованные не себе. Если на роутере не включить:
net.ipv4.conf.all.forwarding = 1
то пакет от office1Server до 8.8.8.8 дойдёт до office1Router, тот увидит что адрес назначения не его — и молча дропнет пакет. Включили на всех 4 роутерах.

Особенности реализации в Proxmox
Linux Bridge без физических портов — виртуальные свитчи внутри гипервизора. ВМ подключаются к ним как к обычной сети, но физически трафик никуда не выходит — только между ВМ на одном хосте.
Статические IP на mgmt-интерфейсе — на Ubuntu 24.04 netplan apply переполучает DHCP и ВМ меняет адрес, теряя SSH. Решение — статика везде.
systemd-сервис вместо if-pre-up.d — старый механизм восстановления iptables не работает на Ubuntu 24 (нет ifupdown). Создали systemd-сервис который запускается до поднятия сети.
MAC-фиксация ens24 — шестой интерфейс centralRouter попал в другой PCI-слот и получил имя enp2s1. Зафиксировали через match: macaddress в netplan чтобы имя всегда было предсказуемым.

Итого — путь пакета от office2Server до 8.8.8.8
office2Server   192.168.1.2
      ↓ default gw 192.168.1.1
office2Router   192.168.255.6
      ↓ default gw 192.168.255.5
centralRouter   192.168.255.2
      ↓ default gw 192.168.255.1
inetRouter      192.168.0.130
      ↓ NAT MASQUERADE → src IP = 192.168.0.130
      ↓ default gw 192.168.0.32
Internet        8.8.8.8
Именно это и показал traceroute в финальной проверке — 4 хопа внутри лаборатории, затем провайдер и Google.
